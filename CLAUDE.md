## High‑Level Code Architecture

- **Project layout** – The repository follows the classic Django layout:
  - `django_project/` – Holds project‑wide settings, URL configuration, WSGI/ASGI entry points.
  - `accounts/`, `doctors/`, `patients/`, `pages/` – Four Django apps that encapsulate distinct domain concerns.
  - `templates/` – Global template directory for shared base templates and static pages.
  - `static/` – Contains custom CSS/JS assets used across the site.
  - `media/` – Runtime upload location for profile pictures and other user‑generated files.

- **Custom User Model** – Defined in `accounts/models.py` as `CustomUser`. It extends `AbstractBaseUser` and adds a `user_type` field (Patient, Doctor, Admin). All authentication flows (sign‑up, login) reference this model.

- **Authentication & Permissions** – Uses Django’s built‑in `django.contrib.auth`. Email‑based login is configured in `django_project/settings.py`. Role‑based access is enforced in view functions and templates via `request.user.user_type` checks.

- **App responsibilities**
  - **accounts** – Handles user registration, login, and role‑based sign‑up forms. Provides context processors for user data.
  - **doctors** – Manages doctor profiles, including specialization, verification status, profile picture upload (`media/`), and CRUD views (`doctor_list`, `doctor_detail`, `doctor_form`).
  - **patients** – Manages patient profiles (medical info) and associated forms.
  - **pages** – Static informational pages (e.g., landing, about) and the dashboard placeholders.

- **URL routing** – Each app defines its own `urls.py`. The project’s root `django_project/urls.py` includes these via `include()`. Namespacing (`app_name`) allows reverse URL lookups like `accounts:login`.

- **Templates** – Uses Django’s templating engine. Base layout resides in `templates/base.html`. App‑specific templates extend this base and are organized under `templates/<app>/` (e.g., `templates/doctors/doctor_list.html`).

- **Media handling** – Profile picture uploads are stored under `media/`. `django_project/settings.py` configures `MEDIA_ROOT` and `MEDIA_URL`. Development server serves media via `django.conf.urls.static.static`.

- **Testing** – Test modules live alongside each app (`tests.py`). The test suite covers model validation, form handling, view permissions, and template rendering. Running `python manage.py test` discovers all `*_tests.py` files automatically.

- **Future work** – The `README.md` notes pending features such as the appointment booking system and user dashboard. Those components will likely introduce new apps (e.g., `appointments`) and additional URL namespaces.

## Development Tips for Claude Code

1. **When adding new views or models, update the corresponding app’s `urls.py` and `admin.py` to keep the project discoverable.**
2. **If you create a new app, remember to add it to `INSTALLED_APPS` in `django_project/settings.py`.**
3. **Static assets are plain CSS/JS; use the existing `static/` directory for additions and reference them via `{% static "path" %}` in templates.**
4. **Media uploads require the `media/` directory to exist; ensure it is present before running the server locally.**
